Configuring Event Console

Audible Alarms for Event Console

Background on Event Console Push Updating

The Event Console application subscribes to the event topic server using the specifications for the current view to specify the events that it wants to be notified about. The topic server then places the list of events onto the JMS queue for the particular console session. Periodically, at intervals defined by three directives, the console application picks up the list of topics and issues the corresponding web services calls to Foundation to get the events which are then displayed. The three operative directives and their function is as follows:

  1. Check interval defines the frequency at which the Event Console reads the topic server message queue. This is currently set to ten seconds.

  2. Batch size is the number of topic server messages which if met or exceeded causes the Event Console to issue the web service request for all of the topic listed since last request which would result in a batch of events being requested that is at least 50 or more since this directive is currently set to 50 events.

  3. Interval size is the time interval at the expiration of which all of the event topics are requested regardless of the number of them. It is currently set to 30 seconds. When this interval is reached, the Event Console requests all of the events for the topics on the topic server queue even if there are less than 50.

The purpose of this arrangement is to arrive at a configurable compromise between latency and performance of the user interface.

Functionality of the Audible Alarm Process (AAP)

Purpose

The purpose of the AAP is to attract the attention of a busy or distracted operator to look at the Event Console. At the same time, it is important not to create alarm fatigue by barraging the operator with an incessant sequence of low priority alarms. The following description of operation results in a balance between these competing objectives.

The AAP scans through the events for each batch update received from Foundation via web services. It selects the highest priority event in each batch and sounds an alarm on the local browser system for a defined interval or until the operator clicks the Silence Alarm button on the Event Console display. The Silence button will stop the alarm if it is sounding, but it will not result in acknowledgement of the alarm. Acknowledging the alarm will also not stop the alarm sounding. There will be no alarm generated for the other events that are in the same batch of events as the highest priority event nor will the alarm sound for a longer time if there are multiple events sharing the highest priority in the batch.

Preferences and Configuration Options

  1. Alarm selection - The System Administrator can choose up to seven different alarm sounds corresponding to the seven different levels of alarm severity in the Foundation database. If desired, the same alarm tone can be selected for all levels of severity. If no alarm is chosen for a particular severity level, no alarm will be sounded for that level. (The reason that this is a system wide selection is to avoid confusing NOC operators by having different tones conveying different meaning on different browsers within earshot of each other.).

  2. The GroundWork Monitor Enterprise Edition will ship with seven different alarm tones available in the .wav directory. The System Administrator can replace these files if desired.

Flexibility

  1. The system can be turned off by the system administrator by not selecting .wav files for the alarm selection directive.

  2. The operator can quiet the alarm by selecting the Silence Alarm button.

  3. The system administrator can have all event generate the same tone, or have a different tone for each of the alarm severities, or by not configuring all of the lower severity events, have alarms only sound for the higher severity events.

  4. Obviously alarms are sounded on the browser only for events that the Event Console on that browser is currently configured to get, i.e. post filters, post custom view tabs, etc.

Turning On the Audible Alarm Functionality

The system alarm functionality is ON by default as seen here in the console.properties file:

#  enable /disable alarm

console.enable.audible.alarms=true

A user can turn it OFF by setting the console.enable.audible.alarm property to equal false as shown here:

#  enable /disable alarm

console.enable.audible.alarms=false

See the Bookshelf document Console Properties to view the entire console.properties file.
