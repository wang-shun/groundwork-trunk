This page presents the backup and restore processes for the Foundation database, JBoss framework, GroundWork Insight Reports, and Configuration (monarch). The operational databases used by GroundWork Monitor are GWCollageDB (Foundation) which contains state and event data, and the Dashboard database which contains historical information used by Insight and Availability Reports.

Foundation Database (GWCollageDB)

This section covers backing up and restoring the Foundation database. Additionally, steps to delete the entire GWCollageDB database and reset the initial settings.

Backup the Foundation Database

You can manually backup the GWCollageDB database at a command line with the following steps:

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

  1. Issue the following command to create a backup sql file. Note: You will be prompted for a password for the postgres user. In the example below, gwcollagedb_backup_09012009.sql is the database backup file name where 09012009 is the current date.

pg_dump -f /usr/local/groundwork/backup/gwcollagedb-11172011.sql.tar -F t -c -E LATIN1 gwcollagedb

  1. Next, restart gwservices enter the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

Restore the Foundation Database

To restore a database that was previously backed up, use the following procedure.

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

  1. Change the directory to the foundation/backup with the following command:

cd /usr/local/groundwork/foundation/backup

  1. You should see a file with the name:

GWCollageDB_backup_<timestamp>.sql

  1. Restore the old database by entering the following command at the Operating System prompt:

pg_restore -d gwcollagedb -F t -c /usr/local/groundwork/backup/gwcollagedb_backup_<timestamp>.sql.tar

  1. Restart gwservices with the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. After relaunching your browser, you should clear your browser's cache.

Delete and Reset the Foundation Database

To delete the entire GWCollageDB database and reset the initial settings, use the following procedure.

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

Set the environment variables for the shell.

source /usr/local/groundwork/scripts/setenv.sh

  1. Change the directory with the following command. This directory has all the database create SQL statements.

cd /usr/local/groundwork/core/databases/postgresql

  1. Drop and re-create the current GWCollageDB database by executing the following script:

psql -f “create-fresh-gwcollagedb.sql” postgres

  1. Load the content of GWCollageDB.sql into GWCollageDB database with the following command:

psql -f “GWCollageDB.sql” postgres

  1. Load all the seed scripts:

psql -f "/usr/local/groundwork/databases/postgresq/postgres-xtra-functions.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-Console.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-Console.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-Metadata.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-State.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-Version.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/GWCollage-PerformanceLabelData.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/foundation-base-data.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/nagios-properties.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/system-properties.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/snmp-properties.sql" gwcollagedb postgres

psql -f "/usr/local/groundwork/databases/postgresq/syslog-seed.sql" gwcollagedb postgres

  1. Restart gwservices with the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. After relaunching your browser, you should clear your browser's cache.

JBoss Databases

The JBoss Portal databases contain information for system users, roles, applications and permissions.

Backup the JBoss Databases

You can manually backup the JBoss databases at a command line with the following steps:

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

Stopping httpd with the command:

/usr/local/groundwork/ctlscript.sh stop apache

Issue the following command to create a backup sql file. Note: You will be prompted for a password for the postgres user. In the example below, jbossdb_backup_09012009.sql is the database backup file name where 09012009 is the current date.

pg_dump -f /usr/local/groundwork/backup/jbossportal_backup_09012009.sql.tar -F t -c -E LATIN1 jbossportal

pg_dump -f /usr/local/groundwork/backup/jbossdb_backup_09012009.sql.tar -F t -c -E LATIN1 jbossdb

  1. To restart gwservices enter the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. Restart httpd with the following command:

/usr/local/groundwork/ctlscript.sh start apache

Restore the JBoss Database

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

Stopping httpd with the command:

/usr/local/groundwork/ctlscript.sh stop apache

  1. Change the directory to backup/jboss with the following command:

cd /usr/local/groundwork/backup/jboss

  1. You should see a file with the name:

jbossportal_backup_<timestamp>.sql

jbossdb_backup_<timestamp>.sql

  1. Drop and recreate the current JBoss databases with the following script:

psql -f “create-fresh-jbossdb.sql” postgres

  1. Restore the old database by entering the following commands at the Operating System prompt:

pg_restore -d jbossportal -F t -c jbossportal_backup_<timestamp>.sql.tar

pg_restore -d jbossdb -F t -c jbossdb_backup_<timestamp>.sql.tar

  1. Restart gwservices with the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. Restart httpd with the following command:

/usr/local/groundwork/ctlscript.sh start apache

  1. After relaunching your browser, you should clear your browser's cache.

Configuration Database (monarch)

This section covers backing up and restoring the Monarch database from the user interface, a command line.

Backup from the User Interface

You have the option to backup the Configuration database via the interface every time a Commit is executed. When you enter the Control>Commit option from the Configuration navigation screen, you will be presented with the options to Abort, Backup or Commit. Selecting the Backup option will create a backup of the Monarch database and Nagios configuration files in the /usr/local/groundwork/core/monarch/backup directory. After the backup completion message is shown, you can select Abort if you do not want to follow through with a Commit.

Backup from a Command Line

You can manually backup the monarch database at a command line with the following steps:

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

Stopping the Nagios deamon with the command:

/usr/local/groundwork/ctlscript.sh stop nagios

Stopping httpd with the command:

/usr/local/groundwork/ctlscript.sh stop apache

Deleting temporary session files with the commands:

rm -rf /tmp/sess*

rm -rf /tmp/tpl*

  1. Issue the following command to create a backup sql file. Note: You will be prompted for a password for the postgres user. In the example below, monarch_backup_09012009.sql is the database backup file name where 09012009 is the current date.

pg_dump -f /usr/local/groundwork/core/monarch/backup/monarch_backup_09012009.sql.tar -F t -c -E LATIN1 monarch

  1. Restart gwservices with the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. Restart httpd with the following command:

/usr/local/groundwork/ctlscript.sh start apache

  1. Restart Nagios with the following command:

/usr/local/groundwork/ctlscript.sh start nagios

Restore the Configuration Database

  1. Open a terminal session and login as: root

  2. Make sure no one is logged into the system by:

Stopping gwservices with the command:

/usr/local/groundwork/ctlscript.sh stop gwservices

Stopping the Nagios deamon with the command:

/usr/local/groundwork/ctlscript.sh stop nagios

Stopping httpd with the command:

/usr/local/groundwork/ctlscript.sh stop apache

Deleting temporary session files with the commands:

rm -rf /tmp/sess*

rm -rf /tmp/tpl*

  1. Change the directory to:

cd /usr/local/groundwork/core/monarch/backup

  1. You should see a file with the name:

monarch_backup_<timestamp>.sql

  1. Restore the old database by entering the following command at the system prompt:

pg_restore -d monarch F t -c monarch_backup_<timestamp>.sql.tar

  1. Restart gwservices with the following command:

/usr/local/groundwork/ctlscript.sh start gwservices

  1. Restart httpd with the following command:

/usr/local/groundwork/ctlscript.sh start apache

  1. Restart Nagios with the following command:

/usr/local/groundwork/ctlscript.sh start nagios

  1. After relaunching your browser, you should clear your browser's cache.

  2. Next, to view and confirm the restored configuration, use the web interface to commit the restored monarch database: Configuration>Control>Pre flight test. And then, Commit>Backup>Commit.
