Acknowledging Problems

  • How To: Acknowledge a troubled host

    [click the how to link again to close this window]

    When a host problem is acknowledged, future notifications about problems are temporarily disabled until the host changes state (i.e. recovers).

    There are a few ways you can acknowledge a troubled host. From the Status application you can locate the host group of the host in trouble or locate the host directly and then acknowledge the problem. Additionally, you can directly acknowledge a host problem from a Dashboard that is setup with the Host List portlet for the troubled host. We'll start our examples below with steps on how to acknowledge a host problem from the host group level, then from the host level, and lastly from a dashboard.

    Acknowledge Problems from the Host Group Level

    1.    In Status, to acknowledge a host problem from the host group level, select the host group (e.g. Linux Servers) of the troubled host (e.g. lt-rbarrysmith1) from the Tree View navigation.

    2.    On the right side of the display, select the Acknowledge link in the Hosts List portlet.

    3.    A popup screen will be displayed, check the Acknowledge this host services too? box if you want to acknowledge all of this hosts troubled services at the same time, check the Send Notification box if you would like the contacts for this host to receive a notification about the acknowledgement.

    4.    Next, check the Persistent Comment checkbox if you would like the host comment to be retained between restarts of Nagios.

    5.    In the Comment field, enter your name and fill in a brief description of your actions.

    6.    Select Submit.

    Figure: Acknowledging a Host from the Host Group level

    Acknowledge Problems from the Host Level

    When selecting a troubled host directly from the Tree View navigation, you can select Acknowledge from the Actions portlet or you can select the Acknowledge link within the Host Information portlet to acknowledge the host.

    Figure: Acknowledging a Host from the Host level

    Acknowledge Problems from a Dashboard

    Alternatively, a troubled host can also be acknowledged directly form a dashboard containing the Host List portlet of the host in trouble.

    Figure: Acknowledging a Host from a Dashboard

    4.    Next, check the Persistent Comment checkbox if you would like the service comment to be retained between restarts of Nagios.

Scheduling Downtime

  • How To: Schedule downtime for hosts and services

    [click the how to link again to close this window]

    Downtime can be scheduled for all hosts in a specified host group, all services in a particular service group, or for a particular host or service. During the specified downtime, Nagios will not send notifications out about hosts or services. This is useful in the event that you actually know you're going to be taking a server down for an upgrade, etc. Scheduling downtime also avoids alarm fatigue, provides more accurate reports, and reinforces change control discipline. When the scheduled downtime expires, Nagios will send out notifications for the hosts and services as it normally would. Scheduled downtimes are preserved across program shutdowns and restarts.

    Schedule Downtime for all Hosts or Services in a Host Group

    1.    In Status, to schedule downtime for all hosts or services in a host group, first select the host group from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Downtime drop-down and select Schedule downtime for all hosts or Schedule downtime for all services.

    3.    A popup screen will be displayed, in the Comment field, enter your name and fill in a brief description of your actions.

    4.    Next, enter a Start Time and End Time for the downtime in the following format: mm/dd/yyyy hh:mm:ss.

    5.    Select the Type of downtime; Fixed or Flexible. If you select the fixed option, the downtime will be in effect between the start and end times you specify. If you do not select the fixed option, Nagios will treat this as flexible downtime. Flexible downtime starts when a host goes down or becomes unreachable (sometime between the start and end times you specified) and lasts as long as the duration of time you enter. The duration fields do not apply for fixed downtime.

    6.    Note that scheduling downtime for services does not automatically schedule downtime for the hosts those services are associated with. If you want to also schedule downtime for all hosts in the host group, check the Schedule downtime for hosts too option.

    7.    Select Submit.

    Figure: Scheduling Downtime for All Hosts or All Services in a Host Group

    Schedule Downtime for a Particular Host

    To schedule downtime for a particular host, select the host form the Tree View navigation, and select either the Actions portlet Downtime drop-down, or the Schedule link in the Host Information portlet.

    Figure: Scheduling Downtime for a Host

    Schedule Downtime for a Particular Service

    Similar to scheduling downtime for a host, to schedule downtime for a particular service, select the service form the Tree View navigation (Hosts or Services tab), and select either the Actions portlet Downtime drop-down, or the Schedule link in the Service Information portlet.

    Figure: Scheduling Downtime for a Service

Enabling, Disabling, and Delaying Notifications

  • How To: Setting host group and service group notifications

    [click the how to link again to close this window]

    When notifications are enabled for all hosts in a specified host group, notifications will only be sent out for the host state types defined in the host definitions. Similarly for services, when notifications are enabled for all services in a service group, notifications will only be sent out for the service state types defined in the service definitions.

    Disabling host group - hosts and services prevents notification from being sent out for all hosts or services in the specified host group or service group.

    Enabling and Disabling Host Group - Host Notifications

    1.    In Status, to enable or disable notifications for all hosts in a host group or all services in a service group, select the host group or the service group from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Notifications drop-down and select Enable or disable notifications for all hosts or Enable or disable notifications for all services.

    3.    A popup screen will be displayed, select Submit. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current notifications status.

    Figure: Enabling Notifications for all Host Group Hosts

     

  • How To: Setting host and service notifications

    [click the how to link again to close this window]

    When notifications are enabled for a host, notifications will only be sent out for the host state types defined in the host definition. Similarly for services, when notifications are enabled for a service, notifications will only be sent out for the service state types defined in the service definitions. Disabling host or service notifications prevents notifications from being sent out for the specified host or service. You will have to re-enable notifications for a host or service before any alerts can be sent out in the future. You can enable and disable notifications for all services on a specific host. Additionally, host and service notifications can also be delayed a specified number of minutes.

    Delaying, Enabling, Disabling Host Notifications

    1.    In Status, to delay the next notification, enable or disable host notifications, or to enable or disable notifications for all host services, select the host from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Notifications drop-down and make your selection, a corresponding popup screen will be displayed;

    Delaying Notifications: This command is used to delay the next problem notification that is sent out for the specified host. The notification delay will be disregarded if the host changes state before the next notification is scheduled to be sent out. This command has no effect if the host is currently UP.

    Disabling/Enabling Host Notifications: These commands are used to prevent and enable specified host notifications. Note that this command does not disable or enable notifications for services associated with this host. Disabling and enabling of host notifications can also be done from the Host Information portlet as seen in our example below.

    Enabling/Disabling Notifications for all Host Services: These commands are used to enable and disable notifications for all services on the specified host. Notifications will only be sent out for the service state types you defined in your service definition. This does not enable notifications for the host unless you check the Enable for host too option.

    3.    Select Submit. for each option. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current notifications status.

    Figure: Enabling Notifications for a Host

    Delaying, Enabling, Disabling Service Notifications

    As in Host Notifications above, to delay the next notification, or enable / disable a service notification, select the service from the Tree View navigation (Host or Service tabs). In the Actions portlet, click the Notifications drop-down and make your selection, a corresponding popup screen will be displayed. Disabling and enabling of service notifications can also be done from the Service Information portlet as seen in our example below.

    Figure: Enabling Notifications for a Service

     

Enabling and Disabling Active Checks

  • How To: Enable, disable active checks for all host group or service group - services

    [click the how to link again to close this window]

    Nagios is capable of monitoring hosts and services in two ways: actively and passively. Active checks are the most common method and are initiated by the Nagios process and run on a regularly scheduled basis. Active checks are initiated by the check logic in the Nagios daemon. When Nagios needs to check the status of a host or service it will execute a plugin and pass it information about what needs to be checked. The plugin will then check the operational state of the host or service and report the results back to the Nagios daemon. Nagios will process the results of the host or service check and take appropriate action as necessary (e.g. send notifications, run event handlers, etc). Active checks are executed at regular intervals, as defined by the check_interval and retry_interval options in your host and service definitions and also on-demand as needed.

    Enabling, Disabling Active Checks for All Host Group or Service Group - Services

    1.    In Status, to enable or disable active checks for all host group or service group services, first select the host group or service group from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Settings drop-down and select Enable active checks for all services or Disable active checks for all services.

    3.    A popup screen will be displayed, this command will enable active checks of all services in the specified host group or service group. Note, for host groups, if you also want to enable active checks of the hosts in the host group check the Enable for hosts too option.

    4.    Select Submit. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current active checks status.

    Figure: Enabling Active Checks for all Host Group Services

    Figure: Nagios Monitoring Statistics - Active Checks

Enabling and Disabling Passive Checks

  • How To: Enable, disable passive checks for a host or service

    [click the how to link again to close this window]

    Nagios is capable of monitoring hosts and services in two ways: actively and passively. In most cases you'll use regularly scheduled active checks. Active checks can be used to poll a device or service for status information every so often. Nagios also supports a way to monitor hosts and services passively instead of actively. They key features of Passive checks are that they are initiated and performed external applications/processes and passive check results are submitted to Nagios for processing. The major difference between active and passive checks is that active checks are initiated and performed by Nagios, while passive checks are performed by external applications. Passive checks are useful for monitoring services that are asynchronous in nature and cannot be monitored effectively by polling their status on a regularly scheduled basis and located behind a firewall and cannot be checked actively from the monitoring host.

    Enabling, Disabling Passive Checks for a Host

    1.    In Status, to enable or disable passive checks for a host, first select the host from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Settings drop-down and select Enable passive checks to allow Nagios to accept passive host check results that it finds in the external command file for this host., or select Disable passive checks to stop Nagios from accepting passive host check results that it finds in the external command file for this host. All passive check results that are found for this host will be ignored.

    3.    Select Submit. Enabling and disabling a host check can also be done from the Host Information portlet using the Check Attempt link, as seen in our example below. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current passive check status.

    Figure: Enabling Passive Checks for a Host

    Enabling, Disabling Passive Checks for a Service

    As in Host passive checks above, to enable or disable passive checks for a service, first select the service from the Tree View navigation. (Host or Service tabs). In the Actions portlet, click the Settings drop-down and make your selection. Enabling passive checks allows Nagios to accept passive service check results that it finds in the external command file for this service. Disabling passive checks stops Nagios from accepting passive service check results that it finds in the external command file for this service. All passive check results that are found for this service will be ignored.

    Figure: Enabling Passive Checks for a Service

     

Enabling and Disabling Flap Detection

  • How To: Enable, disable flap detection for hosts and services

    [click the how to link again to close this window]

    Nagios supports optional detection of hosts and services that are flapping. Flapping occurs when a service or host changes state too frequently, resulting in a storm of problem and recovery notifications. Flapping can be indicative of configuration problems (i.e. thresholds set too low), troublesome services, or real network problems. Whenever Nagios checks the status of a host or service, it will check to see if it has started or stopped flapping. It does this by storing the results of the last 21 checks of the host or service, analyzing the historical check results and determine where state changes/transitions occur, using the state transitions to determine a percent state change value (a measure of change) for the host or service, and comparing the percent state change value against low and high flapping thresholds. A host or service is determined to have started flapping when its percent state change first exceeds a high flapping threshold. A host or service is determined to have stopped flapping when its percent state goes below a low flapping threshold (assuming that is was previously flapping).

    Enabling, Disabling Flap Detection for a Host or Service

    1.    In Status, to enable or disable flap detection for a host or service, first select the host or service from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Settings drop-down and select Enable or disable flap detection to enable or disable flap detection for this host or service. When enabling, if flap detection is disabled on a program-wide basis, this will have no effect.

    3.    Select Submit. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current flap detection status.

    Figure: Disabling Flap Detection for a Host

     

Starting and Stopping Obsessing

  • How To: Start, stop obsessing over a host

    [click the how to link again to close this window]

    The host command directive, Obsess over host, determines whether or not checks for the host will be obsessed over using the ochp_command defined in the Nagios main configuration. The obsessive compulsive host processor command (OCHP) allows you to specify a command to be run after every host check, which can be useful in distributed monitoring. This command is executed after any event handler or notification command. The command argument is the sort name of a command definition that you define in your object configuration file. The maximum amount of time that this command can run is controlled by the ochp_timeout option. This command is only executed if the obsess_over_hosts option is enabled globally and if the obsess_over_host directive in the host definition is enabled.

    Starting and Stopping Obsessing Over a Host

    1.    In Status, to start or stop obsessing over a host, first select the host from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Settings drop-down and select Start or stop obsessing over this host.

    3.    Select Submit.

Enabling and Disabling Event Handlers

  • How To: Enable, disable event handlers for hosts and services

    [click the how to link again to close this window]

    Event Handlers are optional system commands that are run whenever a host or service state change occurs. An obvious use for event handlers is the ability for Nagios to proactively fix problems before anyone is notified. Some other uses for event handlers include restarting a failed service, entering a trouble ticket into a help desk system, and logging event information to a database. Event handlers are executed when a service or host is in a soft problem state, initially goes into a hard problem state, or initially recovers from a soft or hard problem state.

    There are different types of optional event handlers that you can define to handle host and state changes including global host and service event handler, host-specific event handlers, and service-specific event handlers. Global host and service event handlers are run for every host or service state change that occurs, immediately prior to any host- or service-specific event handler that may be run. You can specify global event handler commands by using the Enable event handler directive in the Nagios Main Configuration page, Configuration>Control>Nagios Main Configuration. Individual hosts and services have their own event handler command that should be run to handle state changes. You can specify an event handler that should be run by using the event_handler directive in your host and service definitions. These host- and service-specific event handlers are executed immediately after the (optional) global host or service event handler is executed.

    Enabling, Disabling Event Handlers for a Host or Service

    1.    In Status, to enable or disable event handlers, first select the host or service from the Tree View navigation.

    2.    On the right side of the display in the Actions portlet, click the Event Handlers drop-down and select Enable or disable event handlers to temporarily prevent Nagios from running the host or service event handler for the host or service.

    3.    Select Submit. The Nagios Monitoring Statistics table at the Entire Network, Host Group, and Service Group level views will be updated with the current event handler status.

    Figure: Disabling Service Event Handler for a Service

  • How To: Enable a connection protocol
