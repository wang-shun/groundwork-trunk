0. Overview

Overview

The NSCA Extension Profiler Tool allows the user to simulate passive checks which are send to Nagios on port 5667, processed by Nagios, and send to Foundation on port 4613. The tool generates service checks which can be OK or CRITICAL. A number of hosts and services must be specified, but also the root of the host name and service name. Thus, for example given n (number of hosts), and m (number of services ) we might specifiy
Host Name = atlantis
Service Name = queen
then the system generates automatically names like:
atlantis1, atlantis2, ..., atlatnisn
queen_1, queen_2, ...,queen_m
Simlarly, we specify an IP Address, like 100.001.210.
The system generates following IP addresses:
100.001.210.1, 100.001.210.2, ..., 100.001.210.n

Because the Profiler Tool builds messages, which are bundled in workloads, which are send repeatedly in the case of the NSCA Extension to Nagios on port 5667, we must specify also Number of batches (how many times a workload is repeatedly send to Nagios), the Interval between batches (waiting time between two batches), and a Threshold (the maximum time limit in seconds it takes a message to arrive in the Foundation storage, typically 5 seconds).

We also specify a NSCA Start value, which can be y or n.
When NSCA Start = y it means that the system will reconfigure Nagios with the new values you specified. Each time a reconfiguration takes places, Nagios configuration files, ../nagios/etc/services.cfg and ../nagios/etc/hosts.cfg are rewritten and the Nagios service is stopped and restarted.

The test you create is saved in the configuration file you specify. By choosing a configuration you can run a test (execute). You can run the same test every time you choose to. To speed up the execution by specifying NSCA Start = n you tell the Profiler Tool to disable the Nagios reconfiguration. Nagios assumes that hosts.cfg and services.cfg are configured correctly.

Also, the user is allowed to simulate a real situation by specifying a percentage of how many times a service changes the status from OK to CRITICAL. The field Toggle the Service Status in % (OK)
does this. For example a value of 70 means that in 70% of the cases the service status is OK.

The installation comes with preinstalled examples (configuration files) which can be changed or executed. By doing so this examples can be expanded to address various needs.

For the case of NSCA Extension tests it is important to check the progress of the tests by observing the /usr/local/groundwork/nagios/var/nagios.log files or the /usr/local/groundwork/foundation/container/logs/profiler.log. This log files help the user to observe the execution of the tests.

The Profiler configuration files are stored in $path_to_webapp/profilersa/xml. There are two installation types:
1. By copying the groundwork-profiler.war in the directory /usr/local/groundwork/foundation/container/webapp/ 2. By unzipping the jetty installation which can run on any operating system.
